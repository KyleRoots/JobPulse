You are assisting on an existing procurement/RFP evaluation application that already uses GPT‑4o for document analysis.
Your task in this phase is planning only: produce a detailed technical plan and architecture for a new financial scoring module for RFP bids. Do not modify any code yet.

1. Business context
We support government/enterprise RFP evaluations that happen in three major phases:

Mandatory / technical pass‑fail

Detailed technical scoring

Financial evaluation and final ranking

This epic is about step 3 – financial evaluation for bids that have already passed technical.
​

Key realities from the domain:

Each RFP has a financial evaluation section (e.g., “Section 7.1 – Financial Evaluation”) that defines the evaluation model.
​

Two main financial models:

Lowest cost: lowest price gets highest score; others receive lower scores based on a formula defined in the RFP.
​

Median‑based: the evaluator computes a median price per category over all technically qualified bids, and then scores each bid’s category prices relative to those medians (e.g., median, median ± 10%, median ± 5%, etc.).
​

Only bids that pass technical screening are evaluated financially. The financial module should assume it receives only “technically qualified” bids.
​

Evaluators must maintain a full audit trail for any future disputes and debriefs:

Medians per category

Raw category prices per bid

Computed scores per category and totals per bid

Reasons why a given bid failed or stopped (e.g., did not pass mandatories)
​

For day‑to‑day UI, evaluators mostly care about a ranked list (e.g., the “top 3” or “top 5” bids), but they must be able to drill down to the full history if needed.
​

We already use GPT‑4o to read RFP documents and responses. For this epic, GPT‑4o will primarily be used to:

Extract the financial evaluation rules from the RFP’s financial section.

Normalize category/price structures if needed.
Deterministic scoring (medians, thresholds, totals) should be conventional code, not left to GPT.
​

2. High‑level goals for this epic
Design a financial evaluation subsystem that:

Ingests an RFP’s financial section and structurally represents its evaluation model (lowest cost vs median, criteria, weights, rules).
​

Accepts a set of technically‑qualified bids, each with per‑category prices, and computes:

Per‑category medians (for median‑based models)

Per‑category scores per bid

Total financial scores per bid

Ranked results and “top N” selection (N configurable)
​

Persists all inputs/outputs needed for an audit trail and downstream reporting.
​

Exposes clean, versionable API endpoints for the rest of the app to trigger evaluation and to fetch results.
​

In this planning pass, define the architecture, data models, scoring algorithms, and API shapes, but do not implement them yet.

4. Functional requirements (financial module)
Design for the following:

Evaluation model detection and representation

Given the RFP financial section text, the system should determine:

Whether the evaluation is lowest cost or median‑based.
​

Any rules like “median minus 10”, “median minus 5”, or similar tiers.
​

Any category‑specific rules (e.g., some categories may be excluded or weighted differently).

Your plan should describe:

A data model for storing the financial evaluation model attached to an RFP (e.g., JSON column vs normalized tables).

How GPT‑4o is invoked to extract this model and how we validate its output before using it.
​

Median‑based scoring

For an RFP configured as median‑based, given a set of technically‑qualified bids:

Compute the median for each financial category across all these bids.
​

For each bid and each category, compute a score based on how that bid’s price compares to the category median, following the RFP’s rules (e.g., full points at median, reduced points at median +10%, etc.).
​

Your plan should:

Specify how to compute medians (including even/odd number of bids).

Describe how to parameterize the scoring formula per RFP (not hard‑coding specific percentages).

Describe how to handle missing or malformed data.

Lowest‑cost scoring

For RFPs that specify a lowest‑cost model:

Identify the lowest acceptable price (per category or overall, depending on the model).
​

Compute scores for other bids relative to that lowest price, according to the rules from the RFP.
​

Your plan should:

Describe how to store the lowest‑cost scoring formula.

Explain how to combine per‑category scores into a total financial score.

Audit trail and storage

For each financial evaluation run, we must store:
​

RFP identifier and version

Evaluation model used (as a snapshot, so that if the RFP changes later, the historic run is still interpretable)

The list of bids included (IDs, vendor names, etc.)

For each bid:

Raw category prices

Per‑category medians (median‑based)

Per‑category scores and total financial score

Status: passed/failed financial evaluation, and reason if failed (including “stopped at mandatories” if this module needs to reflect that).

Design a schema that makes it easy to:

Retrieve a full evaluation for a debrief.
​

Answer “why did bid X not make it into the top N?” quickly.

API surface

Propose REST (or equivalent) endpoints for the rest of the app, such as:

POST /rfp/{rfpId}/financial-evaluations

Input: RFP financial rules (or reference), configuration, and the list of technically‑qualified bids and their prices.

Behavior: triggers GPT‑4o extraction of rules (if needed), computes scores, stores results.

GET /rfp/{rfpId}/financial-evaluations/{evaluationId}

Returns the full evaluation, including medians, scores, and all bids.

GET /rfp/{rfpId}/financial-evaluations/{evaluationId}/top

Query params: n, optional filters.

Returns the top N bids with summary info.

Include considerations for:

Idempotency (re‑running evaluations).

Versioning of evaluation logic over time.
